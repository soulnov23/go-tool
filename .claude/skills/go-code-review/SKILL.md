# Go 工程代码审查 Skill — GAN 三 Agent 对抗架构

借鉴 GAN（生成对抗网络）核心思想，通过 **规划器-生成器-评估器** 三个独立 Agent 协作完成代码审查。评估器与生成器形成对抗循环，持续提升代码质量。

参考标准：Go 官方 Code Review Comments、Google Go Style Guide、Uber Go Style Guide、Effective Go、OWASP Go Secure Coding Practices。

---

## 架构概览

```
用户请求
    ↓
┌─────────┐
│  规划器  │  扩展需求为审查规格（只定方向，不指路径）
│ Planner │  → 输出：审查规格文档
└────┬────┘
     ↓
┌─────────┐
│  评估器  │  独立严格审查代码（首轮）
│Evaluator│  → 输出：审查发现（结构化问题列表）
└────┬────┘
     ↓
  模式判断 ─── 只读模式 ──→ 输出审查报告 → 结束
     │
     │ 修复模式（有 🔴 严重问题）
     ↓
┌─────────┐     ┌─────────┐
│  生成器  │ ←→ │  评估器  │  GAN 对抗循环（最多 3 轮）
│Generator│     │Evaluator│
└─────────┘     └─────────┘
     ↓
  输出最终报告（含修复摘要）→ 结束
```

## 两种工作模式

### 模式判断规则

**修复模式**（审查 + 自动修复）触发条件——用户请求中包含以下任一关键词：
- 中文：修复、自动修复、帮我改、顺便改、改掉、修掉、修正、自动改
- 英文：fix、auto-fix、review and fix、patch

**只读模式**（仅审查出报告）：
- 所有不包含上述修复关键词的请求，默认为只读模式
- 典型触发：code review、代码审查、检查代码质量、做一次审查

## 执行流程

### Phase 1: 规划（Planner Agent）

> 详细指令见 `agents/planner.md`，参考资料见 `references/planner/review-dimensions.md`

启动一个 **subagent**，按以下要求执行规划：

1. **识别项目上下文**（并行执行）：
   - 读取 `go.mod` — 确认模块名、Go 版本、核心依赖
   - 浏览顶层目录结构 — 理解包组织方式
   - 从依赖推断技术栈：Web 框架、日志库、ORM、错误处理风格
   - 采样 3-5 个核心业务文件 — 建立项目主流编码风格基线

2. **确定审查范围**：
   - 用户指定文件/目录 → 仅审查指定范围
   - 用户指定维度（如"检查并发安全"）→ 仅执行对应维度
   - 全量审查（如"做一次 code review"）→ 执行全部维度

3. **输出审查规格**（结构化 Markdown）：
   - 项目信息（模块名、Go 版本、技术栈）
   - 审查范围（文件列表或目录）
   - 选定的审查维度及优先级排序
   - 项目风格基线（命名风格、错误处理方式、日志库和字段风格等）
   - 特别关注点（基于技术栈推断的重点领域）

### Phase 2: 评估（Evaluator Agent — 首轮审查）

> 详细指令见 `agents/evaluator.md`，参考资料见 `references/evaluator/checklist.md` 和 `references/evaluator/security-performance.md`

启动一个 **独立 subagent**，接收规划器的审查规格，执行审查：

1. **加载检查清单**：根据规划器选定的维度，加载对应的检查规则
2. **按维度逐项审查**：对审查范围内的代码进行系统性检查
3. **全局一致性检查**：命名/风格/错误处理/日志写法是否全项目统一
4. **输出审查发现**（结构化列表），每个问题包含：
   - 编号 + 严重程度（🔴 严重 / 🟡 建议 / 🟢 提示）
   - 📍 文件位置（`文件:行号`）
   - 📋 问题描述
   - 📖 规范依据
   - ✅ 修复建议代码（修复前/修复后对比）

### Phase 3: 模式分支

#### 只读模式 → 输出审查报告

将评估器的审查发现格式化为最终报告：

```
# Go 代码审查报告

## 项目信息
- 项目：[模块名]
- Go 版本：[版本]
- 技术栈：[识别到的框架/库]
- 审查范围：[全量/指定文件/指定维度]

## 审查概览
- **整体评价**：[优秀/良好/需要改进/存在严重问题]

| 维度 | 🔴 严重 | 🟡 建议 | 🟢 提示 | 总计 |
|------|---------|---------|---------|------|
（按实际审查维度汇总）

## 🔴 严重问题（必须修复）

**[编号] [问题标题]**
- 📍 位置：`文件:行号`
- 📋 描述：[具体问题描述]
- 📖 规范依据：[引用的规范名称和条款]
- ✅ 修复：
  修复前：
  ```go
  // 原始代码
  ```
  修复后：
  ```go
  // 修复代码
  ```

## 🟡 改进建议（建议修复）
（同上格式）

## 🟢 提示信息（可选优化）
（同上格式）

## 亮点
列出代码中做得好的地方

## 优化建议优先级
| 优先级 | 问题 | 收益 | 难度 | 建议 |
|--------|------|------|------|------|

## 后续建议
工具命令和持续改进建议
```

#### 修复模式 → GAN 对抗循环

当评估器发现 🔴 严重问题时，启动对抗循环：

**循环规则**：
- 最多 3 轮对抗
- 每轮仅处理 🔴 严重问题
- 🟡 建议和 🟢 提示不进入循环，仅在最终报告中列出

**每轮循环**：

1. **生成器修复**（Generator Agent）
   > 详细指令见 `agents/generator.md`，参考资料见 `references/generator/style-rules.md` 和 `references/generator/fix-patterns.md`

   启动 subagent，接收评估器的 🔴 问题列表：
   - 逐个读取问题所在文件
   - 按项目风格基线编写修复代码
   - 使用 Edit 工具应用修复
   - 运行 `go build ./...` 和 `go vet ./...` 验证编译和基本正确性

2. **评估器验收**（Evaluator Agent — 验收轮）
   启动 **新的独立 subagent**（保持信息隔离，不看生成器思考过程）：
   - 重新审查被修复的文件
   - 逐项验证每个 🔴 问题是否已正确修复
   - 检查修复是否引入新问题
   - 输出验收结果：
     - ✅ 通过 → 退出循环
     - ❌ 未通过 → 列出仍未解决的问题 + 新引入的问题

3. **循环结束条件**（满足任一即退出）：
   - 评估器验收通过（所有 🔴 问题已修复且未引入新问题）
   - 达到最大轮次（3 轮）

**输出最终报告**：
- 基于最终状态生成完整审查报告（同只读模式报告格式）
- 额外包含修复摘要：

```
## 修复摘要
- 对抗轮次：[N] 轮
- 🔴 严重问题：[已修复数]/[总数]
- 修复的文件：[文件列表]
- 未解决问题（如有）：[问题列表 + 原因]
```

## 问题严重等级划分标准

| 等级 | 标识 | 范围 | 示例 |
|------|------|------|------|
| 严重 | 🔴 | 正确性缺陷、安全漏洞、数据竞争、资源泄漏、panic 风险、死锁 | 未处理 error、goroutine 泄漏、math/rand 生成密钥、SQL 注入、nil map 写入 |
| 建议 | 🟡 | 风格违规、命名不当、违反最佳实践、可维护性差、日志不规范 | snake_case 命名、未格式化代码、接收者用 this、日志混用多个库 |
| 提示 | 🟢 | 性能优化空间、文档完善、代码简化、内存优化 | 未预分配 slice 容量、缺少导出函数注释、热路径可用 strconv 替代 fmt |

## 全局原则

1. **对抗性**：评估器与生成器独立运作，评估器不妥协，不因"生成器已尝试修复"而降低标准
2. **信息隔离**：评估器验收时只看代码结果，不看生成器的修复思路
3. **自适应**：规划器先识别项目技术栈和主流风格，以此为基准
4. **一致性优先**：同一项目中相同语义的代码必须使用相同写法
5. **可操作**：每个问题必须附带可直接使用的修复代码
6. **务实优先**：只报告真正有价值的问题，避免噪音
7. **高效**：并行工具调用，大项目可按模块分批审查
