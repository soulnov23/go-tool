# Go 工程通用代码审查 Skill

对任意 Go 工程执行全方位代码审查，自适应识别项目技术栈，以项目自身主流风格为基准。

参考标准：Go 官方 Code Review Comments、Google Go Style Guide、Uber Go Style Guide、Effective Go、OWASP Go Secure Coding Practices。

## 审查范围

根据用户请求确定审查范围：

- **指定文件/目录**：仅审查指定范围
- **指定维度**（如"检查并发安全"）：仅执行对应维度
- **全量审查**（如"做一次 code review"）：执行全部维度

## 审查维度（12 大维度）

每个维度的详细检查项见 `references/go-review-checklist.md`，风格与命名的正反代码示例见 `references/go-style-rules.md`，安全/并发/性能规则见 `references/go-security-performance.md`。

> **贯穿所有维度的核心原则**：同一项目中，相同语义的代码必须使用相同的写法。每个维度的审查不仅检查单点是否合规，还要检查全项目内该维度的写法是否统一。

### 基础规范层

1. **命名一致性** — 文件、包、函数、变量、接口、常量、缩写词、接收者命名风格；**全项目同一概念必须使用同一命名**（如 `ctx`/`err`/`req`/`cfg`/`mu`/`logger` 等约定变量名不可混用）
2. **代码风格与可读性** — 格式化、导入分组、行长度、缩进错误流、声明风格、注释文档、裸返回、代码组织；**相同功能的函数调用全项目统一写法**（构造函数、HTTP 响应、类型转换等写法不可混用）
3. **错误处理** — 错误创建与包装、哨兵错误、错误字符串格式、错误检查不遗漏、一次处理原则、带内错误避免；**错误包装方式和错误消息格式全项目统一**

### 逻辑正确性层

4. **资源与生命周期管理** — defer/Close 资源释放、context 使用、goroutine 生命周期、channel 安全
5. **逻辑正确性与 Bug 检测** — 边界条件、nil 安全、切片/map 陷阱、类型断言、defer 陷阱、数据竞争、时间处理
6. **并发安全与死锁** — 锁竞争优化、死锁风险、并发原语选择、goroutine 数量控制、伪共享

### 性能优化层

7. **内存分配与 GC 压力** — 逃逸分析、堆分配优化、sync.Pool、预分配、字符串操作、闭包逃逸
8. **运行时性能** — 热路径优化、strconv 替代 fmt、值传递 vs 指针传递、接收者类型选择

### 安全层

9. **安全编码** — 输入验证、SQL 注入防护、crypto/rand 使用、敏感信息保护、依赖漏洞扫描、文件路径安全、HTTP 安全

### 设计层

10. **类型系统与接口设计** — 接口定义位置、接口大小、编译时接口验证、结构体嵌入、泛型使用、Option 模式
11. **测试质量** — 表驱动测试、测试失败信息、子测试、边界覆盖、基准测试、模糊测试、竞态检测

### 工程规范层

12. **项目工程规范** — 包设计与职责、依赖管理、init() 使用、全局变量、日志规范、构建标签、导出 API 稳定性；**日志库统一、日志字段 key 命名统一、同类操作日志必备字段统一**

## 执行流程

### 1. 识别项目上下文

并行执行以下操作：

- 读取 `go.mod` — 确认模块名、Go 版本、核心依赖
- 浏览顶层目录结构 — 理解包组织方式
- 从依赖推断技术栈：Web 框架、日志库、ORM、错误处理风格
- 采样 3-5 个核心业务文件 — 建立项目主流编码风格基线
- 加载需要的 references 文件

### 2. 按维度审查

核心原则：

- **以项目自身的主流风格为基准**，标记偏离主流风格的代码
- **语义搜索优先**，精确搜索补充，按需深入读取文件
- **只报告有价值的问题**，忽略过于琐碎的风格偏好
- 每个问题提供：文件位置、问题描述、规范依据、修复前后代码对比
- 按严重程度排序：🔴 严重（必须修复）→ 🟡 建议（建议修复）→ 🟢 提示（可选优化）

### 3. 输出审查报告

报告格式：

```
# Go 代码审查报告

## 项目信息
- 项目：[模块名]
- Go 版本：[版本]
- 技术栈：[识别到的框架/库]
- 审查范围：[全量/指定文件/指定维度]

## 审查概览
- **整体评价**：[优秀/良好/需要改进/存在严重问题]

| 维度 | 🔴 严重 | 🟡 建议 | 🟢 提示 | 总计 |
|------|---------|---------|---------|------|
（12 个维度汇总）

### 问题严重等级划分标准

| 等级 | 标识 | 范围 | 示例 |
|------|------|------|------|
| 严重 | 🔴 | 正确性缺陷、安全漏洞、数据竞争、资源泄漏、panic 风险、死锁 | 未处理 error、goroutine 泄漏、math/rand 生成密钥、SQL 注入、nil map 写入 |
| 建议 | 🟡 | 风格违规、命名不当、违反最佳实践、可维护性差、日志不规范 | snake_case 命名、未格式化代码、接收者用 this、日志混用多个库 |
| 提示 | 🟢 | 性能优化空间、文档完善、代码简化、内存优化 | 未预分配 slice 容量、缺少导出函数注释、热路径可用 strconv 替代 fmt |

## 审查原则

1. **自适应**：先识别项目技术栈和主流风格，以此为基准逐项检查
2. **一致性优先**：对于风格类问题，以项目中已确立的多数写法为基准，标记所有偏离基准的代码；同一种操作只允许一种写法
3. **对事不对人**：审查代码质量，而非评判开发者
4. **提供依据**：每个问题引用具体的规范条款
5. **可操作**：每个问题必须附带可直接使用的修复代码
6. **认可优点**：对做得好的地方给予正面反馈
7. **务实优先**：只报告真正有价值的问题，避免噪音
8. **高效**：并行工具调用，大项目可按模块分批审查
9. **分层**：基础规范 → 逻辑正确性 → 性能优化 → 安全 → 设计 → 工程规范
