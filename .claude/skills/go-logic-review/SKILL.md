# Go 业务逻辑审查 — GAN 三 Agent 对抗架构

通过 **规划器-评估器-生成器** 三个独立 Agent 协作，审查并修复业务逻辑缺陷。

```
用户请求
    ↓
┌─────────┐
│  规划器  │ → 审查规格
└────┬────┘
     ↓
┌─────────┐
│  评估器  │ → 逻辑缺陷列表
└────┬────┘
     ↓
  模式判断 ── 只读 ──→ 输出报告 → 结束
     │
     │ 修复（有 🔴 问题）
     ↓
┌─────────┐     ┌─────────┐
│  生成器  │ ←→ │  评估器  │  对抗循环（最多 3 轮）
└─────────┘     └─────────┘
     ↓
  最终报告 → 结束
```

## 模式判断

- **修复模式**：用户请求含「修复/fix/auto-fix/帮我改/改掉」
- **只读模式**：其余情况（默认）

## 执行流程

### Phase 1: 规划器（Planner）

> 指令见 `agents/planner.md`

启动 subagent：分析项目业务上下文，输出审查规格（业务领域、审查范围、关键流程）。

### Phase 2: 评估器（Evaluator — 首轮）

> 指令见 `agents/evaluator.md`

启动独立 subagent：接收审查规格，系统性审查业务逻辑，输出结构化缺陷列表。

每个问题包含：编号 + 严重程度 + 位置 + 描述 + 业务影响 + 触发场景 + 修复建议。

### Phase 3: 分支

**只读** → 格式化输出审查报告。

**修复** → 进入 GAN 对抗循环：

1. **生成器**（`agents/generator.md`）：修复所有 🔴 问题
2. **评估器**（新的独立 subagent，信息隔离）：验收修复结果
3. 循环直到：全部通过 或 达到 3 轮上限

## 严重等级

| 等级 | 标识 | 范围 |
| ---- | ---- | ---- |
| 严重 | 🔴    | 会导致业务结果错误的逻辑缺陷 |
| 建议 | 🟡    | 逻辑不够健壮，存在潜在风险 |
| 提示 | 🟢    | 逻辑可优化 |

## 核心原则

1. **对抗性**：评估器不妥协，不因生成器已修复而降低标准
2. **信息隔离**：评估器验收时只看代码结果
3. **场景驱动**：每个问题必须有具体触发场景
4. **业务优先**：以业务正确性为准则
