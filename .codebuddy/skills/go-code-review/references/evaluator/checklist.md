# Go 代码审查检查清单（评估器参考）

> 供评估器使用：12 维度完整检查清单，含通过/失败判定标准。
>
> 参考标准：Go 官方 Code Review Comments、Google Go Style Guide、Uber Go Style Guide、Effective Go、OWASP Go-SCP
>
> 审查时先识别项目主流风格，以此为基准逐项检查。[严重] 标记为 🔴 必须修复项。

---

## 一、命名一致性

### 检查项

- 文件名：小写 + 下划线 (`snake_case.go`)，测试文件 `_test.go` 后缀
- 包名：全小写、简短、单词不拆分，无下划线，避免 `common`/`util`/`misc`/`base`/`helper`/`api`
- 包名与目录名一致，不使用复数形式
- 避免选择容易被局部变量遮蔽的包名
- 导出标识符：大驼峰 `PascalCase`
- 未导出标识符：小驼峰 `camelCase`（非 `snake_case`，非 `ALL_CAPS`）
- 错误变量：导出 `Err` 前缀，未导出 `err` 前缀；错误类型 `Error` 后缀
- 构造函数：`New` / `NewXxx`，不混用 `Create`/`Make`/`Build`
- Option 函数：`With` 前缀
- 接口：单方法 `-er` 后缀；多方法描述性名词
- 缩写词：全大写或全小写一致（`ID` 非 `Id`，`HTTP` 非 `Http`）
- Getter 不加 `Get` 前缀
- 布尔函数：`Is`/`Has`/`Can` 前缀
- 接收者名称：1-2 字母，不用 `this`/`self`/`me`，同类型一致
- 常量：MixedCaps，不用 `ALL_CAPS`

### 通过条件
- 所有标识符命名符合 Go 惯例
- 同一概念在全项目中使用同一名称

### 失败条件（🔴 严重）
- [严重] 同一概念在不同文件中使用不同变量名（如 `userID` vs `userId` vs `uid`）
- [严重] 包名与目录名不一致
- [严重] 约定变量名（ctx/err/req/cfg/mu/logger）在不同文件中混用不同名称

### 全局一致性检查
- 扫描全项目，确认同一概念命名统一
- 检查接收者名称在同类型方法间一致
- 检查 for range 循环变量命名统一
- 检查 channel 命名风格统一
- 变量声明风格统一（`:=` vs `var`）

## 二、代码风格与可读性

### 检查项

**格式化**：
- 代码通过 `gofmt` / `goimports` 格式化
- 行长度软限制 99 字符
- `if` 条件不换行

**导入分组**：
- 标准库 → 第三方库 → 项目内部包，各组空行隔开
- 避免不必要的导入重命名
- [严重] 禁止 `import .` 形式

**缩进与错误流**：
- [严重] 先处理错误并返回，正常逻辑最小缩进
- 不用 `else` 包裹正常路径
- 提前 return/continue 减少嵌套

**声明风格**：
- 显式赋值用 `:=`，空切片用 `var`（nil 切片），结构体初始化用字段名
- 未导出全局变量加 `_` 前缀

**注释文档**：
- [严重] 所有导出的类型/函数/常量必须有 godoc 注释
- 注释以标识符名称开头，完整句子

**裸返回**：
- 裸返回仅用于极短函数

### 通过条件
- 代码格式统一，导入分组正确
- 错误处理在前，正常流程最小缩进
- 导出标识符都有文档注释

### 失败条件
- [严重] 导出函数缺少 godoc 注释
- [严重] 使用 `import .` 形式
- [严重] 错误处理后用 else 包裹正常路径

### 全局一致性检查
- 构造函数/初始化写法全项目统一（函数式选项 vs 配置结构体）
- HTTP 响应写法统一
- 类型转换写法统一
- 超时设置方式统一

---

## 三、错误处理

### 检查项

**错误创建**：
- 静态错误 `errors.New`，需匹配的用导出 `var ErrXxx`
- 动态错误 `fmt.Errorf`，需匹配的用自定义错误类型

**错误包装**：
- [严重] 需保留错误链用 `%w`，不需要时用 `%v`
- 错误消息添加有意义的上下文
- 使用 `errors.Is()` / `errors.As()` 判断

**错误字符串格式**：
- [严重] 不首字母大写（除专有名词）
- [严重] 不以标点结尾

**错误处理原则**：
- [严重] 不用 `_` 丢弃错误（除非有注释解释）
- [严重] 错误一次处理：不既记录日志又返回
- `os.Exit` / `log.Fatal` 仅在 `main()` 中

### 通过条件
- 所有错误都被正确处理
- 错误包装保留链，消息有上下文
- 错误格式符合 Go 标准

### 失败条件（🔴 严重）
- [严重] 用 `_` 忽略关键函数的错误返回值
- [严重] 错误既记录日志又返回
- [严重] 错误字符串首字母大写或以标点结尾
- [严重] 错误包装混用 `fmt.Errorf` 和 `errors.Wrap`（一致性）
- [严重] 错误消息格式混乱

---

## 四、资源与生命周期管理

### 检查项

- [严重] 文件/连接/事务获取成功后立即 `defer Close()`
- 错误分支正确清理资源
- 循环中不 defer（在循环内清理或提取函数）
- [严重] context 作为函数第一个参数，不存储在 struct 中
- [严重] goroutine 有明确退出机制（context cancel / channel close）
- [严重] channel 避免 send on closed（发送方负责关闭）

### 通过条件
- 所有可关闭资源都有对应的 close/释放
- goroutine 都有可控的退出时间
- context 使用规范

### 失败条件（🔴 严重）
- [严重] 忘记 `defer Close()` 导致资源泄漏
- [严重] goroutine 无退出机制
- [严重] send on closed channel

---

## 五、逻辑正确性与 Bug 检测

### 检查项

- [严重] nil map 写入、nil 指针解引用检查
- 零值、空切片、空字符串、空 map 处理
- 整数溢出/下溢、除零
- [严重] API 边界处复制切片和 map
- [严重] 使用安全类型断言 `value, ok := x.(Type)`
- [严重] 不复制包含 `sync.Mutex` 的结构体
- [严重] 共享变量有正确同步保护
- `sync.WaitGroup.Add` 在 goroutine 启动前调用

### 通过条件
- 无 nil 安全问题
- 类型断言使用安全形式
- API 边界正确防护

### 失败条件（🔴 严重）
- [严重] nil map 写入
- [严重] 不安全的类型断言
- [严重] 复制含 Mutex 的结构体
- [严重] 数据竞争

---

## 六、并发安全与死锁

### 检查项

- 热路径可否用 `atomic` 替代 `Mutex`
- 读多写少用 `sync.RWMutex`
- [严重] 锁范围最小化，不在锁内 I/O
- Mutex 零值有效，不嵌入导出结构体
- [严重] 锁嵌套顺序全局一致
- channel 操作有超时或 default
- [严重] 控制 goroutine 数量（worker pool / semaphore）

### 通过条件
- 锁使用合理，范围最小
- 无死锁风险
- goroutine 数量受控

### 失败条件（🔴 严重）
- [严重] 锁内执行网络/数据库请求
- [严重] goroutine 无限制创建
- [严重] 锁嵌套顺序不一致

---

## 七、内存分配与 GC 压力

### 检查项

- 返回局部变量指针导致逃逸
- `any`/`interface{}` 参数导致装箱
- 小结构体传值而非指针
- 高频临时对象用 `sync.Pool`
- [严重] 已知大小的切片/map 预分配
- 字符串拼接用 `strings.Builder`
- 热路径用 `strconv` 而非 `fmt`
- 闭包不捕获大对象

### 通过条件
- 已知大小的集合都预分配
- 热路径无不必要的分配

### 失败条件
- [严重] 未预分配大切片频繁 append

---

## 八、运行时性能

### 检查项

- 不为节省字节传指针（小结构体传值）
- 接收者类型选择合理
- 同类型不混用值/指针接收者
- 优先同步函数
- 热路径 `strconv` 而非 `fmt`

### 通过条件
- 值/指针传递合理
- 接收者类型一致

---

## 九、安全编码

### 检查项

- [严重] 外部输入在使用前验证
- [严重] 参数化查询，禁止 SQL 拼接
- [严重] `crypto/rand` 生成安全随机数
- [严重] 日志不打印敏感数据
- [严重] 防止路径遍历
- HTTP 超时设置合理
- 限制请求体大小
- 依赖漏洞扫描

### 通过条件
- 无 SQL 注入风险
- 安全随机数正确使用
- 敏感信息不泄露

### 失败条件（🔴 严重）
- [严重] 使用 `math/rand` 生成安全随机数
- [严重] SQL 字符串拼接
- [严重] 敏感信息泄露到日志
- [严重] 路径遍历漏洞

---

## 十、类型系统与接口设计

### 检查项

- [严重] 接口由使用方定义，非实现方
- 接口尽量小（单方法最灵活）
- 编译时接口验证 `var _ Interface = (*Type)(nil)`
- 嵌入字段在顶部，互斥锁不嵌入
- 不使用指向接口的指针
- 序列化字段必须有标签

### 通过条件
- 接口定义位置正确
- 结构体嵌入合理

### 失败条件
- [严重] 在实现方为 mock 定义接口
- [严重] 互斥锁嵌入导出结构体

---

## 十一、测试质量

### 检查项

- 表驱动测试减少重复
- [严重] 失败信息含函数名、输入、got、want
- 子测试 `t.Run()` 组织
- 结构比较用 `cmp.Equal`/`cmp.Diff`
- 核心逻辑有单元测试
- 边界和错误路径有覆盖
- [严重] 并发代码用 `go test -race`

### 通过条件
- 测试失败信息有诊断价值
- 边界条件有覆盖

### 失败条件
- [严重] 未使用 `-race` 运行测试
- [严重] 测试失败信息无具体内容

---

## 十二、项目工程规范

### 检查项

- 每个包职责单一，避免循环依赖
- [严重] `init()` 不执行 I/O、网络、复杂逻辑
- [严重] 避免可变全局变量
- `go.mod` 版本一致，定期 `go mod tidy`
- 使用 `go vet`、`golangci-lint`
- [严重] 日志库统一，日志字段 key 统一
- 错误日志包含足够上下文

### 通过条件
- 无可变全局变量
- 日志库和字段风格全项目统一

### 失败条件（🔴 严重）
- [严重] init() 中执行 I/O
- [严重] 可变全局变量
- [严重] 日志混用多个库
- [严重] 日志字段 key 名不一致

### 全局一致性检查
- 日志库调用方式全项目统一
- 日志消息风格统一（大小写、标点、用词）
- 同类操作日志携带相同上下文字段
- 错误日志统一包含 request_id、业务 ID、error
